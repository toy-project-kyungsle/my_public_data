# P0) 전문성과 안정성

## (1) 시스템 관련

- **현재 에러가 어떤 방향에서 얼마나 많이 나고 있는지를 촘촘히 체크한다**
	- [[앱 안정성 현황 체크]]

- 실패를 설계한다
	- 자동으로 재시도를 해야하는 요청/액션/UI는 무엇이 있을까?
		- [ ] 결제 실패했을 때 정확한 에러메시지나 가이드가 가서, 유저가 다시 구매할 수 있도록 하는가?
		- [ ] 쿠폰 적용 실패했을 때 쿠폰 해제 후 다시 시도하려는 움직임이 있는가?
		- [ ] 로그인을 실패한 유저가 다시 시도할 수 있게 가이드를 제공하는가?
		- [ ] 로그인을 진행하는 중에 내가 로그인을 하고 있다는 것을 제대로 인지할 수 있게 되어 있는가?
	- 장애의 폭발 반경(Blast Radius)을 줄인다
		- [ ] (웹) Error Boundary로 추천/리뷰/배너 영역 격리
		- [ ] (RN) 화면 단위 fallback + 재시도 버튼 + 고객센터/다른 결제수단 이동
		- [ ] (성능) 결제 화면에서 heavy component 지연 로딩/분리
	- 사용자에게 안내가 나가야 하는 요청/액션/UI는 무엇이 있을까?
		- [ ] 사용자에게 에러페이지가 나가고 있는 것들이 어떻게 나가고 있는지 조사하기
		- [ ] 모든 에러가 에러페이지만 보여야 할까? 모달 or 토스트 or 고객센터 인입 등의 방법이 있는데 하나만 쓰는게 아닐까?
	- 보안상이나 발생해서는 안 되는 케이스가 무엇인지 제대로 인지하고 있고 공유되었나?
	- 점진적으로 롤아웃 할 수 있는 설계과 되어 있는가?
	- 복구할 수 있는 포인트가 설계되어 있는가?

- 고위험(결제/쿠폰/로그인/장바구니) 변경 개발에 대해서 롤백/플래그/완화 전략이 있는가?
	- [ ] 1%→10%→50% 단계적 배포 (카나리 등)
	- [ ] 결제 이탈률/에러율/타임아웃이 임계치 넘으면 자동/수동 OFF
	- [ ] 웹과 앱 환경에서 롤백이라는 것을 어떻게 하면 빠르게 할 수 있을까?

- 위 성과들은 사실 눈에 띠지 않는다.
	- 이 것들이 발생할 수 있었는데 나의 코드적인 대비와 시스템적 설계로 막았다는 것을 어떻게 알것인가?

## (2) 코드 관련

- **타입**을 강하게 사용해서 코딩 에러를 방지하기
	- [ ] 타입을 강하게 만들려면 어떤 것을 할 수 있는지 조사해보기
	- [ ] 앱에 있는 타입 중에서 안정성이 중요한 타입에 대해서 우선순위 매기기. 리스트업해서 하나씩 제거하기.
	- [ ] zod를 더 잘 사용하기. 누가 zod를 작성하더라도 제대로 작성이 되는 시스템 만들기
	- [ ] Union Type을 적용해본다
	- [ ] Brand 타입을 적용해본다

- **assert**와 **guard**함수를 사용해서 절대 발생하지 않아야 하는 것들을 체크한다
	- [ ] 우리 앱에 적용한다면 어디에 적용할 수 있을지 고민해보기. 아마 핵심 퍼널이라고 보인다.

- 절대로 일어나서는 안 되는 것들 **테스트 코드** 작성하기
	- 절대로 일어나서는 안 되는 것이란 무엇인가?
		- [ ] 결제 상태는 무조건 단방향으로 가야한다. 대비가 되어 있는가?
		- [ ] 주문은 반드시 한 번만 생성된다고 확신할 수 있는가?
		- [ ] 쿠폰이 중복 적용 불가능한가? 쿠폰 적용 로직이 깨끗하게 관리되고 있는가?
		- !! 사실 너무 다양하다 이건 추후에 더 아이디어를 내어도 될 것 같다

- 일어나면 UX에 영향을 주는 것들 테스트 코드 작성하기
	- [ ] 결제 중인 제품은 해당 제품을 다른 사람이 구매할 수 없도록 booking 되어 있는가?
	- [ ] 장바구니 담기 했던 상품의 가격이 달라진 경우에 제대로 된 알림을 보여주는가?
	- [ ] [[Storybook을 통해서 P0 기여하기]] : 예측 가능한 컴포넌트 만들기. 프론트 어드민부터 하면 좋아보인다.

- 테스트 코드를 또 잘 작성하려면 함수나 변수에 대한 **의존성 분리**가 잘 되어 있어야 한다
	- [ ] 함수는 어떻게 분리하는 것이 좋은 것인가?
	- [ ] 단위 테스트를 잘 할 수있는 함수란 어떤 것인가?
	- [ ] 하나의 역할을 하는 함수와 변수를 어떻게 만들 수 있는가?
	- [ ] [[레이어와 위험한 코드 격리]]
	- 좋은 코드에 대한 공부 진행하기
		- [ ] 준프책을 읽으면서 활용해볼 수 있는 것들이 무엇이 있을지 고민해보기

- **lint**를 사용해서 혹시 모를 에러를 방지한다. lint도 사용하자면 사용할 수 있는 방법이 무궁무진 할 것 같다.
	- ✅ [[api 파일에서 import 순환 참조 문제 - 원인 및 인사이트 도출]] : lint에 순환참조 관련한 warning을 추가했다.


- 에러가 안 나게(또는 나도 안전하게) 만드는 설계/코딩 습관을 갖추려면 무엇을 해야 하는가?
	- [ ] 안정성에 초점을 맞추어서 개발을 하는 가이드/영상/책이 찾아서 읽어보기
	- [ ] use effect 에 잘못된 값이 들어가는 것에 대한 강력한 경고. 
		- 두 번이나 이 것 때문에 큰 에러가 발생하거나 발생할 뻔 했다.



# P0) AI를 잘 사용하기

- **클로드 코드 잘 사용하기**
	- 전략  AI → 시니어 리뷰 AI
		-  🔥[[AI 에이전트와 스킬 공유]]
	- AI의 역할을 분리해라: 작성자 ≠ 리뷰어 ≠ 테스터 ≠ 릴리즈 매니저
	- “명세/불변식 → 실패 시나리오 → 구현” 순서를 고정하라
	- KPI 과제에서는 “가설”보다 “측정 설계”를 먼저 시켜라
	- 클린 코드에 대한 가이드를 AI에게 제공하기
	- “보안/데이터 노출”은 지침으로 선제 봉인해라. ENV나 KEY들
	- "프롬프트/체크리스트를 코드처럼 버전 관리”하라
	- 슈퍼클로드에서 각자의 커맨드가 어떤 역할을 하는지 알아보고 싶다

- AI가 일정한 수준의 코드를 작성해내기
	- 우리가 어떤 코드를 작성하고 있는지에 대한 규칙을 정할 필요가 있다
	- 애슬러 앱이나 어드민에 대한 코드 가이드를 작성해본다.

- 핵심 플로우(결제/쿠폰/장바구니/로그인)에서 AI 쓸 때
	- AI가 생각했을 때 절대 바뀌어서는 안 되는 '불변식'이 무엇인지 뽑아보게 하기
		- 커머스는 로직 정합성이 핵심이라, AI에게 바로 코드를 시키면 사고가 난다.
		- 쿠폰 적용 후 결제금액은 0 미만 불가
		- 결제 완료 이벤트는 주문번호 기준 중복 발화 금지
		- 장바구니 수량/가격은 서버 확정 응답 이후만 확정 렌더

- AI에게 가설 생성 및 검증 방법을 맡기기
	- 예: “검색 전환율을 올리고 싶다” → AI가
		- 이벤트 정의(검색→상품노출→상세→장바구니→결제)
		- 어떤 로그/필드를 추가해야 원인을 분해할지
		- 실험군/대조군, 성공지표, 가드레일 지표(예: 결제 이탈률 악화 방지)

- "CTR은 올랐는데 구매는 왜 안 오르지?" 와 같은 의문점을 찾고 AI에게 이 것을 조사시켜보기
