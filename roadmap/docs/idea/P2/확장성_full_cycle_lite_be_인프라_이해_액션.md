> **프론트가 BE/인프라를 "실무 임팩트(안정성/전환/KPI/운영)"를 내기 위해 이해하는 능력 — 성능/장애 원인을 서버까지 분해하고, 관측/완화하고, 비용 감각을 갖추는 것**

아래는 바로 실행 가능한 플랜(2주/6주/3개월) + 매일/매주 루틴 + "프론트가 주도할 수 있는 BE/인프라 과제" 리스트야.

---

## 0) 목표 정의(프론트 관점의 BE/인프라 식견이란)

프론트가 BE/인프라를 "잘 안다"는 건 결국 4가지를 할 수 있는 상태야.

### 액션

1. **성능/전환 이슈를 '서버까지' 원인 분해**한다 (네트워크/캐시/DB/큐/레이트리밋)
2. **장애를 재현/완화**한다 (타임아웃, 재시도, 서킷브레이커, 폴백, 롤백)
3. **관측 가능하게 만든다** (로그/메트릭/트레이스, 코릴레이션 ID)
4. **비용/리소스를 감각적으로 본다** (트래픽/페이로드/쿼리/캐시 히트율 → 비용)

✅ Done: BE/인프라 식견의 4가지 목표가 정의됨.

## 1) 2주 액션플랜: "네트워크/관측"을 내 손에 넣기

### 액션

**A. 네트워크 계약(Contract) 읽고 쓰기**

- 자주 쓰는 API 5개 선정 → **요청/응답 스키마, 에러코드, 타임아웃, idempotency(멱등성)** 문서화
- RN/웹 공통으로 **API 호출 규약** 정리:
  - 타임아웃 기본값(예: 7~10초) + 화면별 override
  - 재시도 조건(네트워크 단절/5xx만, 4xx는 X)
  - 백오프(지수 백오프 + jitter)
  - 중복 클릭/중복 요청 방지(멱등키 or client request id)

**B. "프론트 → 서버" 추적 가능하게 만들기 (코릴레이션)**

- 모든 API 요청 헤더에 `X-Request-Id`(uuid) 넣기
- 서버 로그에 그 값이 찍히도록 BE와 합의(가능하면 응답에도 echo)
- 에러 리포팅(Sentry 등)에 request-id, endpoint, status, RTT, retry_count 포함

✅ Done: `api-contract.md` + 공통 fetch/axios wrapper 개선 PR 1개 + 장애/CS 재현 시 "서버 로그까지 바로 찾는" 루트 확보.

## 2) 6주 액션플랜: "장애 대응력"을 시스템으로 만들기

### 액션

**A. 실패 모드(Failure modes) 카탈로그 만들기**

패션 커머스에서 프론트가 자주 만나는 실패 10개만 뽑아서, **증상→원인 후보→확인법→완화책**을 정리해.

**예시**

- 결제 직전 API 504
  - 원인 후보: upstream 지연/DB lock/서드파티 지연/ALB idle timeout
  - 확인: 서버 p95/p99, 특정 endpoint slow log, 외부 PG latency
  - 완화: 프론트는 재시도 금지(중복결제), 폴백 UI, 상태조회(polling) 전환

**B. "프론트 주도"로 BE/인프라 개선 1개 끝내기**

아래 중 하나를 골라 1개는 반드시 **측정→개선→재측정**까지.

- **캐시/압축**: 이미지/JSON payload gzip/br, 캐시헤더 정리(ETag/Cache-Control)
- **레이트리밋/봇 방어**: 과도한 호출 화면(검색/필터) 디바운스 + 서버 rate limit 협의
- **N+1 호출 제거**: 화면 진입 시 5~10번 호출 → BFF(Backend For Frontend)로 합치기 제안
- **서킷브레이커/폴백**: 추천/리뷰 등 외부성 강한 API는 실패 시 기본 UI로 degrade

✅ Done: `frontend-failure-playbook.md` (온콜 없어도 대응 가능한 수준) + "p95 TTFB -20%" "에러율 -30%" "전환 +0.x%" 같이 숫자로 남김.

## 3) 3개월 액션플랜: "비용/확장/운영" 감각 만들기

### 액션

**A. 트래픽 비용을 프론트가 줄이는 프로젝트**

- **앱 번들/웹 번들 최적화**: 코드 스플릿, 이미지 포맷(webp/avif), 폰트 서브셋
- **서버 비용 절감**: payload 줄이기(필요 필드만), pagination, 캐시 히트율 높이기
- **데이터 비용 절감**: 이벤트 로깅 샘플링/중복 제거, 과다 이벤트 정리

**B. 인프라 운영 지표를 "내 지표"로 보기**

- 주 1회 30분: CloudWatch(또는 대시보드)에서
  - API p95/p99, 5xx, 타임아웃, ALB Target 5xx
  - DB 커넥션/CPU, 큐 적체(있다면), 배포 직후 에러 스파이크
- 프론트 릴리즈와 서버 지표가 같이 움직이는지 "연결"해서 보는 습관이 핵심

✅ Done: "릴리즈 체크리스트 + 대시보드 링크 + rollback 기준" 문서 1개.

## 4) 매일/매주 실행 지침(가볍지만 누적되는 루틴)

### 액션

**매일(10~15분)**

- 오늘 내가 만든/수정한 기능의 **API 1개**를 골라서
  - status code, 실패 시 UI, retry 정책이 합리적인지 점검
  - "이 요청이 p95 1초면 UX가 버틸까?"를 한 번 생각

**매주(60분)**

- 지난주 장애/슬로우/CS 이슈 중 1개를 골라
  - "클라 로그 → 서버 로그"로 추적 리포트 1장 작성
  - 다음 주 액션 1개(관측/완화/성능/계약 변경)로 연결

✅ Done: 매일/매주 루틴이 정착됨.

## 5) BE/인프라 식견을 빨리 올리는 "학습 순서"

공부도 하되, 범위를 좁혀서 "내가 매일 만지는 것"부터.

### 액션

1. **HTTP/캐시/압축/CORS** (프론트 즉효)
2. **타임아웃/재시도/멱등성/서킷브레이커** (장애 대응 즉효)
3. **CDN/이미지 파이프라인** (커머스 전환 즉효)
4. **DB/인덱스/슬로우쿼리 개념** (원인 분해용)
5. **큐/비동기/이벤트 드리븐** (주문/알림/재고 등 흐름 이해)
6. **배포/롤백/카나리** (운영 임팩트)

✅ Done: 학습 우선순위가 정해지고, 순서대로 실행됨.

## 6) "프론트가 BE/인프라와 협업"을 잘하는 커뮤니케이션 템플릿

BE/인프라에 제안할 때는 이 4줄이면 거의 먹혀.

### 액션

- **현상**: 어떤 화면에서 무엇이 느리거나 깨진다 (재현 steps)
- **데이터**: p95/p99, 에러율, 영향 사용자/전환
- **가설**: 네트워크/서버/DB/외부 연동 중 어디일 가능성이 큰지
- **요청**: 로그 필드 추가 / 캐시헤더 / 엔드포인트 합치기 / 타임아웃 조정 등 "한 번에 할 수 있는 최소 변경"

✅ Done: BE/인프라 협업 커뮤니케이션 템플릿이 정착됨.
